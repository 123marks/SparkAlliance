# 星火联盟 - 设置板块深度完善方案

> 本文档基于对项目代码库的逆向分析，指出原设计方案中**缺失的关键环节**，并给出可直接落地的补全方案。  
> 核心目标：让设置模块从"功能清单"升级为"可工程化交付的闭环系统"。

---

## 零、原方案问题诊断（必读）

在深入阅读原方案和逆向整个 `frontend/src` 后，我发现以下 **结构性缺陷**：

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| 1 | **设置没有独立路由** | P0 | 侧边栏"设置"链接指向 `/app/profile`，而 Profile.vue 是个人中心页，不是设置页。两者职责完全不同。 |
| 2 | **没有 user_settings 数据表** | P0 | DATA-MODELS.md 里没有设置相关表结构。设置数据无处存放。 |
| 3 | **没有 useSettings composable** | P0 | 项目每个模块都有对应的 composable（useShop / useHealth / useSchedule 等），唯独设置没有。 |
| 4 | **存储分层策略缺失** | P0 | 原方案没有讨论哪些设置存 localStorage（外观）、哪些存 Supabase（隐私/安全）、哪些存 user_metadata（昵称）。混存会导致数据丢失或同步冲突。 |
| 5 | **Supabase RLS 策略缺失** | P0 | 设置表需要行级安全策略，确保用户只能读写自己的设置。原方案完全没提。 |
| 6 | **设置变更的跨模块联动缺失** | P1 | 改了"免打扰"，SparkMessages 模块怎么知道？改了"主题色"，全局 CSS 变量怎么更新？没有事件机制。 |
| 7 | **离线策略缺失** | P1 | 外观设置必须在无网络时也能生效。原方案假设始终在线。 |
| 8 | **默认值与迁移策略缺失** | P1 | 老用户没有 settings 记录怎么办？settings schema 升级怎么办？ |
| 9 | **Supabase Auth user_metadata 未充分利用** | P1 | 现有代码已经用 `user.user_metadata.nickname` 取昵称，但原方案忽略了这个现成能力。 |
| 10 | **过度设计** | P2 | "AI 生成头像"、"社交关系图谱"、"情绪日记"等功能本身就是独立产品，不应放在设置模块的迭代计划里。 |

## 一、架构补全：路由 + 布局 + 导航

### 1.1 路由设计

**当前问题**：`router/index.ts` 中没有 `/app/settings` 路由，侧边栏设置链接错误指向 `/app/profile`。

**修复方案**：

```
路由拓扑：
/app/settings                → 设置入口，redirect 到 /app/settings/profile
/app/settings/profile        → 个人资料
/app/settings/security       → 账号安全
/app/settings/privacy        → 隐私设置
/app/settings/notifications  → 通知设置
/app/settings/appearance     → 外观主题
/app/settings/features       → 功能设置
/app/settings/data           → 数据存储
/app/settings/about          → 关于帮助
```

**路由配置（需加入 router/index.ts）**：

```typescript
// 在 /app children 中新增：
{
  path: 'settings',
  component: () => import('../pages/app/Settings.vue'),
  redirect: '/app/settings/profile',
  children: [
    { path: 'profile', name: 'SettingsProfile', component: () => import('../pages/app/settings/ProfileSettings.vue') },
    { path: 'security', name: 'SettingsSecurity', component: () => import('../pages/app/settings/SecuritySettings.vue') },
    { path: 'privacy', name: 'SettingsPrivacy', component: () => import('../pages/app/settings/PrivacySettings.vue') },
    { path: 'notifications', name: 'SettingsNotifications', component: () => import('../pages/app/settings/NotificationSettings.vue') },
    { path: 'appearance', name: 'SettingsAppearance', component: () => import('../pages/app/settings/AppearanceSettings.vue') },
    { path: 'features', name: 'SettingsFeatures', component: () => import('../pages/app/settings/FeatureSettings.vue') },
    { path: 'data', name: 'SettingsData', component: () => import('../pages/app/settings/DataSettings.vue') },
    { path: 'about', name: 'SettingsAbout', component: () => import('../pages/app/settings/AboutSettings.vue') },
  ]
}
```

**侧边栏修正（AppLayout.vue）**：

```
当前：router-link to="/app/profile"  → 设置
修正：router-link to="/app/settings" → 设置
保留：router-link to="/app/profile"  → 个人中心（从下拉菜单进入）
```

### 1.2 设置页与个人中心的职责边界

| 维度 | 个人中心 Profile.vue | 设置中心 Settings.vue |
|------|---------------------|----------------------|
| 入口 | 顶部头像下拉 → 个人中心 | 侧边栏底部 → 设置 |
| 职责 | 展示个人主页：头像、动态、热力图、统计 | 修改所有配置项：资料、安全、外观等 |
| 读/写 | 以"读"为主，仅"编辑资料"按钮跳转设置 | 以"写"为主，所有表单在这里 |
| 路由 | /app/profile | /app/settings/* |

**关键联动**：Profile.vue 的"编辑资料"按钮应 `router.push('/app/settings/profile')`，形成闭环。

## 二、数据层补全：存储分层 + 数据库 + RLS

### 2.1 设置数据存储分层策略

这是原方案最大的缺失。不同类型的设置数据必须存在不同的地方：

```
┌─────────────────────────────────────────────────────┐
│                  设置数据存储分层                       │
├──────────────┬──────────────────────────────────────┤
│ 存储层        │ 存储内容                               │
├──────────────┼──────────────────────────────────────┤
│ Supabase     │ nickname, realName, gender, birthday  │
│ Auth         │ location, bio（通过 user_metadata）    │
│ user_metadata│ 优势：注册时就能写入，Auth API 自带     │
├──────────────┼──────────────────────────────────────┤
│ Supabase     │ education, privacy, notifications,    │
│ user_settings│ features, recommendation              │
│ 表           │ 需要服务端校验和 RLS 保护的数据         │
├──────────────┼──────────────────────────────────────┤
│ localStorage │ appearance（theme, fontSize,          │
│ + 内存缓存    │ primaryColor, animationEnabled）      │
│              │ 必须离线可用、即时生效的数据             │
├──────────────┼──────────────────────────────────────┤
│ Supabase     │ 密码修改、邮箱绑定、手机绑定、          │
│ Auth API     │ 第三方 OAuth、账号删除                  │
│ (直接调用)    │ 通过 supabase.auth.updateUser 等       │
└──────────────┴──────────────────────────────────────┘
```

### 2.3 Row Level Security (RLS) 策略

```sql
-- 启用 RLS
ALTER TABLE user_settings ENABLE ROW LEVEL SECURITY;

-- 用户只能读自己的设置
CREATE POLICY "Users can read own settings"
  ON user_settings FOR SELECT
  USING (auth.uid() = user_id);

-- 用户只能更新自己的设置
CREATE POLICY "Users can update own settings"
  ON user_settings FOR UPDATE
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

-- 用户可以创建自己的设置（首次）
CREATE POLICY "Users can insert own settings"
  ON user_settings FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- 用户可以删除自己的设置（注销时）
CREATE POLICY "Users can delete own settings"
  ON user_settings FOR DELETE
  USING (auth.uid() = user_id);
```

## 三、核心遗漏：useSettings Composable

这是将设置数据与 UI 连接的关键枢纽。项目中其他模块都有对应 composable，设置模块不能没有。

### 3.1 设计原则

```
useSettings 的职责：
1. 启动时加载设置（localStorage + Supabase 双源）
2. 提供响应式 settings 对象供组件绑定
3. 提供 updateSettings() 方法，自动判断存储目标
4. 设置变更时发出事件，通知其他模块
5. 处理默认值填充、数据校验、错误恢复
```

### 3.2 核心接口设计

```typescript
// composables/useSettings.ts

interface UseSettingsReturn {
  // 状态
  settings: Ref<UserSettings>          // 完整设置，响应式
  appearance: Ref<AppearanceSettings>  // 外观设置，独立管理
  isLoading: Ref<boolean>
  isSaving: Ref<boolean>
  error: Ref<string | null>
  
  // 方法
  loadSettings: () => Promise<void>    // 初始化加载
  updateProfile: (data: Partial<ProfileData>) => Promise<void>
  updatePrivacy: (data: Partial<PrivacyData>) => Promise<void>
  updateNotifications: (data: Partial<NotificationData>) => Promise<void>
  updateAppearance: (data: Partial<AppearanceSettings>) => void  // 同步, 即时生效
  updateFeatures: (data: Partial<FeaturesData>) => Promise<void>
  resetSection: (section: SettingsSection) => Promise<void>
  exportSettings: () => Promise<Blob>
  
  // 事件
  onSettingsChange: (callback: (section: string, value: any) => void) => void
}
```

### 3.3 关键实现逻辑

```
加载流程：
1. 先从 localStorage 读取外观设置 → 立即应用到 CSS 变量（确保首屏不闪烁）
2. 再从 Supabase Auth 读取 user_metadata → 填充 profile 数据
3. 再从 Supabase user_settings 表读取其余设置
4. 合并所有数据，用默认值填充缺失字段
5. 如果 user_settings 表没有记录 → 自动创建一条（upsert）

保存流程：
updateProfile() → supabase.auth.updateUser({ data: { nickname, ... } })
updatePrivacy() → supabase.from('user_settings').update({ privacy: ... })
updateAppearance() → localStorage.setItem() + 更新 CSS 变量 + 发事件
updateNotifications() → supabase.from('user_settings').update({ notifications: ... })

错误处理：
- 网络失败 → 用 localStorage 缓存写操作，恢复网络后重试
- 数据损坏 → 回退到默认值，log 错误
- 并发冲突 → updated_at 乐观锁，冲突时取服务端最新值
```

## 四、跨模块联动机制

### 4.1 问题描述

当用户在设置中修改了某个配置，其他模块需要实时响应：

| 设置变更 | 受影响模块 | 预期行为 |
|---------|-----------|---------|
| 开启免打扰 | SparkMessages, 通知系统 | 停止弹窗和声音提醒 |
| 切换深色/浅色主题 | 全局 | CSS 变量即时切换 |
| 修改主题色 | 全局 | 品牌色系即时更新 |
| 关闭粒子效果 | CosmicBackground.vue | 停止粒子渲染 |
| 修改 AI 默认模型 | Chat.vue | 下次对话使用新模型 |
| 修改日程提醒时间 | Schedule 模块 | 更新提醒策略 |
| 修改隐私可见性 | Profile, CampusWall | 内容展示权限变化 |

### 4.2 解决方案：基于 Vue Reactive + Event Bus

```
方案选择理由：
- 不引入额外依赖（不用 mitt / EventEmitter）
- 利用 Vue 3 的 reactive 本身就是全局响应式的特性
- useSettings 中的 ref 是模块级单例（文件顶层声明），所有组件共享同一个引用
- 对于需要副作用的场景（如更新 CSS 变量），使用 watch

实现方式：
1. useSettings 导出的 appearance 是同一个 ref（模块级单例）
2. 任何组件 import { useSettings } 拿到的是同一份数据
3. 在 App.vue 或 AppLayout.vue 中 watch appearance，驱动全局副作用
```

### 4.3 CSS 变量同步机制

```typescript
// 在 AppLayout.vue 或 App.vue 中：
const { appearance } = useSettings()

watch(appearance, (newVal) => {
  const root = document.documentElement
  root.style.setProperty('--color-brand-blue', newVal.primaryColor)
  root.style.setProperty('--font-size-base', `${newVal.fontSize}px`)
  
  // 主题切换
  if (newVal.theme === 'light') {
    root.classList.add('theme-light')
    root.classList.remove('theme-dark')
  } else {
    root.classList.add('theme-dark')
    root.classList.remove('theme-light')
  }
  
  // 动画控制
  if (!newVal.animationEnabled) {
    root.style.setProperty('--animation-duration', '0s')
  }
}, { deep: true, immediate: true })
```

## 五、默认值与数据迁移策略

### 5.1 问题

现有用户（已注册、已使用的用户）在 `user_settings` 表中没有记录。当设置模块上线后，这些用户打开设置页会怎样？

### 5.2 方案：按需创建 + 默认值填充

```
流程：
1. useSettings.loadSettings() 被调用
2. 查询 user_settings WHERE user_id = current_user
3. 如果没有记录 → upsert 一条，所有字段用默认值
4. 如果有记录但缺少某些字段（schema 升级后）→ 用 JS 层面的 deepMerge 填充默认值
5. 返回合并后的完整设置对象
```

### 5.3 深度合并函数

```typescript
function deepMergeDefaults<T extends Record<string, any>>(
  data: Partial<T>,
  defaults: T
): T {
  const result = { ...defaults }
  for (const key of Object.keys(defaults)) {
    if (data[key] !== undefined && data[key] !== null) {
      if (typeof defaults[key] === 'object' && !Array.isArray(defaults[key])) {
        result[key] = deepMergeDefaults(data[key], defaults[key])
      } else {
        result[key] = data[key]
      }
    }
  }
  return result
}
```

## 六、离线策略与首屏防闪烁

### 6.1 问题

用户打开应用时，Supabase 请求还没返回，页面应该显示什么主题？如果用的是浅色主题但默认是深色，就会出现"闪一下"的问题。

### 6.2 方案

```
时序：
1. index.html 加载
2. <script> 标签（在 Vue 挂载前）立即读取 localStorage 中的外观设置
3. 将 CSS 变量注入到 <html> 的 style 属性
4. Vue 应用挂载，CosmicBackground 等组件使用已正确设置的 CSS 变量
5. Supabase 请求完成后，其他设置就绪

关键：步骤 2-3 必须在 Vue 挂载前执行，这意味着需要在 index.html 中加一段同步 JS。
```

```html
<!-- index.html 中，在 <script type="module" src="/src/main.ts"> 之前 -->
<script>
  // 首屏防闪烁：在 Vue 加载前应用外观设置
  try {
    const stored = JSON.parse(localStorage.getItem('spark-appearance') || '{}')
    const root = document.documentElement
    if (stored.theme === 'light') {
      root.classList.add('theme-light')
    }
    if (stored.primaryColor) {
      root.style.setProperty('--color-brand-blue', stored.primaryColor)
    }
    if (stored.fontSize) {
      root.style.setProperty('--font-size-base', stored.fontSize + 'px')
    }
  } catch (e) { /* 忽略解析错误，使用默认主题 */ }
</script>
```

## 七、安全设计补全

### 7.1 原方案遗漏

原方案中"账号与安全"章节设计了 UI，但没有说明：
- 密码修改需要 Supabase Auth 的哪个 API
- 邮箱/手机绑定需要什么后端逻辑
- 登录设备管理需要什么数据表
- API Key 加密存储的具体方式

### 7.2 密码修改流程（Supabase 实现）

```typescript
// 修改密码 - 需要用户已登录
async function changePassword(newPassword: string): Promise<void> {
  const { error } = await supabase.auth.updateUser({
    password: newPassword
  })
  if (error) throw error
  // 注意：Supabase 不支持验证"旧密码"
  // 如需此功能，需要 Edge Function 封装
}
```

**关键问题**：Supabase Auth 的 `updateUser` 不验证旧密码。如果需要这个安全特性，必须：

```
方案 A（推荐）：
1. 用户输入旧密码
2. 前端用旧密码调用 supabase.auth.signInWithPassword() 验证
3. 验证通过后再调用 supabase.auth.updateUser({ password: newPassword })

方案 B：
使用 Supabase Edge Function 封装，在服务端完成整个流程
```

### 7.3 API Key 存储安全

原方案提到用户可以添加自己的 AI API Key。这是高敏感数据。

```
存储方案：
1. API Key 不能明文存在前端 localStorage
2. 不能明文存在 user_settings 表（即使有 RLS，数据库管理员仍可见）
3. 应使用 Supabase Vault（如果可用）或 Edge Function 做中转

简化方案（适合当前阶段）：
1. API Key 存在 user_settings.features -> ai -> apiKeys 中
2. 前端显示时脱敏（sk-****...abc）
3. 使用时由 Edge Function 读取并转发请求
4. RLS 确保只有用户自己能读取

后续进阶：
- 使用 pgcrypto 扩展对 API Key 加密存储
- 或使用 Supabase Vault secret 管理
```

## 八、设置搜索功能

### 8.1 为什么需要

设置中心有 8 个分类、50+ 个设置项。用户找一个特定设置（比如"关闭粒子效果"）需要逐个翻分类。这是糟糕的体验。

### 8.2 方案：前端本地搜索

```typescript
// 设置项索引，用于搜索
interface SettingsSearchItem {
  id: string           // 唯一标识
  section: string      // 所属分类 key
  sectionLabel: string // 分类名称
  label: string        // 设置项名称
  keywords: string[]   // 搜索关键词（包括别名、拼音首字母等）
  path: string         // 路由路径
}

const settingsIndex: SettingsSearchItem[] = [
  {
    id: 'theme-mode',
    section: 'appearance',
    sectionLabel: '外观主题',
    label: '深色模式',
    keywords: ['深色', '暗色', '夜间', 'dark', 'theme', '主题', '模式'],
    path: '/app/settings/appearance'
  },
  {
    id: 'particle-effect',
    section: 'appearance',
    sectionLabel: '外观主题',
    label: '粒子效果',
    keywords: ['粒子', '背景', '动画', 'particle', '星空', '特效'],
    path: '/app/settings/appearance'
  },
  {
    id: 'dnd',
    section: 'notifications',
    sectionLabel: '通知设置',
    label: '免打扰模式',
    keywords: ['免打扰', '静音', '勿扰', 'dnd', '通知', '安静'],
    path: '/app/settings/notifications'
  },
  // ... 所有设置项
]

// 搜索函数
function searchSettings(query: string): SettingsSearchItem[] {
  const q = query.toLowerCase().trim()
  if (!q) return []
  return settingsIndex.filter(item =>
    item.label.toLowerCase().includes(q) ||
    item.keywords.some(kw => kw.toLowerCase().includes(q))
  )
}
```

### 8.3 搜索 UI

设置页顶部放置搜索框，输入时实时过滤，点击结果项跳转到对应分类并高亮该设置项。

## 十二、被原方案遗漏但极其重要的细节

### 12.1 设置修改的"保存"策略

```
方案 A：自动保存（推荐）
- Toggle/Select 类控件：修改即保存，显示"已保存"微提示
- Input 类控件：失去焦点时保存
- 优点：减少用户操作步骤
- 缺点：网络慢时可能误操作

方案 B：手动保存
- 底部固定"保存"按钮
- 优点：用户可以批量修改后一次性保存
- 缺点：忘记保存导致修改丢失

推荐：外观设置用自动保存（即时反馈），安全设置用手动保存（防误操作）
```

### 12.2 设置修改的 Undo 机制

```
实现：
1. 每次 updateSettings 前，snapshot 当前值
2. 保存成功后，显示 Toast："设置已更新 [撤销]"
3. 点击撤销 → 恢复 snapshot 并再次保存
4. Toast 5 秒后自动消失
```

### 12.3 浏览器通知权限

```
通知设置中"浏览器推送"需要额外处理：
1. 检查 Notification.permission 状态
2. 如果是 'default' → 显示"请求权限"按钮
3. 如果是 'denied' → 显示"请在浏览器设置中开启"
4. 如果是 'granted' → 显示开关

// 请求权限
async function requestNotificationPermission() {
  const result = await Notification.requestPermission()
  return result === 'granted'
}
```

### 12.4 数据导出格式

```
导出内容（JSON 格式）：
{
  "version": "1.0",
  "exportedAt": "2026-03-30T12:00:00Z",
  "user": {
    "nickname": "...",
    "email": "...",
    "createdAt": "..."
  },
  "settings": { ... },
  "posts": [ ... ],
  "likes": [ ... ],
  "schedule": [ ... ]
}

实现：
1. 前端发请求到 Supabase Edge Function
2. Edge Function 查询该用户所有数据
3. 打包成 JSON
4. 返回下载链接（Supabase Storage 临时 URL，24h 有效）
```

## 十三、关键决策点总结

以下决策会影响整个设置系统的架构，需要在开发前确定：

| 决策点 | 选项 A | 选项 B | 建议 |
|--------|--------|--------|------|
| 设置保存策略 | 自动保存 | 手动保存 | 按分类：外观自动，安全手动 |
| 外观设置存储 | 全存 localStorage | 存 Supabase + 本地缓存 | localStorage，不需同步 |
| API Key 存储 | 前端加密存储 | 服务端 Edge Function | 先方案 A，后续迁移方案 B |
| 设置搜索 | 前端静态索引 | 后端 ElasticSearch | 前端索引，50 个设置项不需要后端 |
| 通知推送 | Web Push API | 仅应用内通知 | 先应用内，后续加 Web Push |
| 设备管理 | 自建追踪表 | 依赖 Supabase Auth sessions | 先跳过，后续自建 |
