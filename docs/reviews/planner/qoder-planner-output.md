## 与日程模块的边界定义

### 本质区别：一个管方向，一个管路径

| 维度 | 智能日程 (Schedule) | 星火规划 (Planner) |
|------|---------------------|-------------------|
| **核心问题** | "什么时候做什么？" | "要达成什么？做到什么程度？" |
| **思维模型** | 时间轴驱动 | 目标驱动 |
| **数据粒度** | 具体时间段的事件 | 可量化的目标与任务 |
| **时间跨度** | 以天/周为主视角 | 以周/月/学期为主视角 |
| **完成标准** | 事件发生即完成 | 需要主动标记或量化达标 |
| **AI角色** | "这个时间段你有空，建议安排在这里" | "根据你的目标，你应该做这些，按这个节奏" |
| **情感体验** | 有序感、掌控感 | 成就感、成长感 |

### 日程已覆盖（规划不做）

- 日历视图（月/周/日）
- 事件 CRUD + 颜色标签分类
- 课表导入（教务系统/OCR/手动）
- 重复事件管理
- AI 智能时间推荐（基于空闲时段）
- 冲突检测与解决
- 事件提醒

### 规划独占功能（日程不涉及）

- 目标设定与层级拆解（愿景 → 关键结果 → 每日任务）
- 完成率追踪与进度可视化
- 习惯养成（连续打卡、累计打卡、补签机制）
- 游戏化激励（星空领域、经验值、等级、成就徽章）
- AI 长期目标拆解与动态调节
- 社交激励（好友互督、集体目标、成就分享）
- 成长数据统计与回顾报告

### 联动方案（协同而不重叠）

```
联动方向：单向推送，规划 → 日程

用户在"规划"中设定：考研数学每天做2小时真题
    ↓
规划模块产出任务："做2024年数学真题卷一"
    ↓
用户可选择"安排到日程"
    ↓
日程模块自动创建事件："数学真题 14:00-16:00 图书馆3B"
    ↓
日程完成后状态回传给规划（标记任务完成）
```

**数据关联方式**：规划任务表中保留 `schedule_event_id` 字段，日程事件表中已有 `related_task_id` 字段（MODULE-SCHEDULE.md 第498行），天然支持双向关联。

### 一句话定义

> **规划是方向盘和仪表盘，日程是导航仪——规划告诉你"去哪里、走了多远"，日程告诉你"下一步走哪条路"。**

## 目标体系设计方案

### 层级设计：2.5 层架构

经过权衡，推荐"两层为主、三层可选"的弹性架构：

```
第一层：目标 (Goal)
├── 示例："考研上岸985" / "四级600+" / "减重10kg"
├── 属性：类型、截止日期、量化指标、优先级
├── 状态：进行中 / 已完成 / 已放弃 / 已暂停
│
├── 第1.5层：关键结果 (Key Result) ← 可选，复杂目标才需要
│   ├── 示例："数学140+" / "英语65+" / "政治70+"
│   └── 每个关键结果有独立进度条
│
└── 第二层：任务 (Task)
    ├── 示例："做线代第三章习题" / "背50个单词" / "跑步5公里"
    ├── 属性：所属目标、日期、完成状态、重复规则
    └── 完成任务 → 进度值累加到上级目标
```

**为什么不直接三层？** 大学生的多数目标（四六级、每日健身、阅读30本书）只需要"目标 → 任务"两层，强制三层增加认知负担。但"考研"这类多科目大目标确实需要中间层，因此设计为可选展开。

**MVP 先做两层**（Goal + Task），关键结果层在 V2 加入。

### 目标类型矩阵

| 类型 | 典型场景 | 量化方式 | 预设模板 |
|------|---------|----------|---------|
| **学业目标** | 考研、GPA提升、四六级、考证 | 分数/等级 | 考研倒计时、四六级冲刺、期末GPA |
| **技能目标** | 编程(LeetCode)、设计、外语、写作 | 完成数/熟练度 | LeetCode百题计划、英语口语30天 |
| **习惯目标** | 早起、健身、阅读、冥想、记账 | 打卡天数/累计量 | 早起挑战、阅读马拉松、健身打卡 |
| **自定义目标** | 任何用户自定义 | 用户自定义 | 空白模板 |

### 时间维度策略

| 维度 | MVP | V2 | 说明 |
|------|-----|-----|------|
| **每日** | 支持 | - | 每日任务清单，核心交互入口 |
| **每周** | 支持 | - | 周目标回顾，周报数据来源 |
| **每月** | 支持 | - | 月度目标进度，月报数据来源 |
| **学期** | - | 支持 | 匹配大学生学期节奏，与课表联动 |
| **年度** | - | 支持 | 年度总结，高传播价值 |

### 任务属性设计

```typescript
interface PlannerTask {
  id: string
  goal_id: string                    // 所属目标
  title: string                      // 任务标题
  task_type: 'once' | 'daily' | 'weekly' | 'custom'  // 任务类型
  quantity_target?: number           // 量化目标（如：50个单词、5公里）
  quantity_unit?: string             // 单位（个、公里、小时、页）
  quantity_done?: number             // 已完成量
  scheduled_date?: string            // 计划日期（一次性任务）
  recurrence_days?: number[]         // 重复日 [1,3,5] = 周一三五
  is_completed: boolean
  completed_at?: string
  schedule_event_id?: string         // 关联日程事件ID
  xp_reward: number                  // 完成奖励经验值
}
```

## 游戏化机制设计（优先级排序）

### P0 — MVP 必做

#### 1. 基础星星系统

每创建一个目标，在用户的"个人星空"中生成一颗星星。

```
星星状态演变：

  ☆ 暗淡 (Dim)        → 目标刚创建，尚未开始
  ✧ 闪烁 (Flickering)  → 进行中，完成率 1%-49%
  ★ 明亮 (Bright)      → 完成率 50%-99%
  ✦ 璀璨 (Radiant)     → 目标达成 100%
  ✸ 超新星 (Supernova)  → 超额完成或提前完成（彩蛋奖励）
```

- 星星大小由目标周期决定：日目标(小) < 月目标(中) < 学期目标(大)
- 每颗星有独立的"光芒进度条"，完成任务 → 进度增长 → 星星逐渐变亮
- 星空背景为暗色系，与项目"暗黑系 + 星火配色"一致

**视觉设计方向**：

```
┌─────────────────────────────────────┐
│           我的星空                    │
│                                     │
│        ✦                ★          │
│     "四级600+"       "每日健身"      │
│                                     │
│              ✧                      │
│           "考研上岸"                 │
│                                     │
│    ☆                    ✦          │
│  "Python进阶"       "阅读20本"       │
│                                     │
│  ─────────────────────────────────  │
│  已点亮 3/5    总光芒值: 2,450       │
└─────────────────────────────────────┘
```

#### 2. 每日打卡机制

- 进入规划页面后，"今日任务"列表是主交互区
- 完成任务 → 勾选 → 播放微动效（星光粒子飞向对应星星）
- 所有今日任务完成 → 触发"今日全勤"动画
- 打卡记录两个计数器：**连续天数** + **累计天数**（断签不清零累计值）

#### 3. 断签保护设计

这是留存率的关键。断签不能让用户觉得"前功尽弃"。

```
策略组合：

1. 双计数器
   - "连续天数"归零，但"累计天数"永不减少
   - 用户看到"虽然连续断了，但我已经坚持了47天"，心理损失更小

2. 补签卡
   - 每月自动发放2张免费补签卡
   - 可补最近7天内的断签
   - 使用补签卡恢复连续天数

3. 宽限期
   - 当日23:59前未打卡 → 次日12:00前仍可补打（算前一天）
   - 超过宽限期才真正断签

4. 冰冻模式
   - 用户可提前设置"暂停日"（如：考试周、旅行）
   - 暂停期间不计入连续天数，但也不断签
   - 每月最多冰冻3天

5. 回归奖励
   - 断签后回归，前3天完成任务给予1.5倍经验值
   - "回归挑战"：连续3天完成 → 恢复断签前50%的连续天数
```

## AI规划助手设计

### 核心定位

与日程模块的 AI 功能做明确区分：

| 能力 | 日程AI | 规划AI |
|------|--------|--------|
| 时间推荐 | "你14:00-16:00有空" | 不涉及 |
| 冲突检测 | "这个时间已有课" | 不涉及 |
| **目标拆解** | 不涉及 | "考研可以分3个阶段" |
| **任务生成** | 不涉及 | "今天建议做线代第3章" |
| **难度调节** | 不涉及 | "你最近完成率低，减少20%" |
| **进度诊断** | 不涉及 | "照这个速度你可能来不及" |

### AI 动态调节机制

```
触发条件              → AI 行为
───────────────────────────────────────
7天完成率 < 50%       → 弹出提醒："最近有点吃力？"
                        建议：减少每日任务量20%
                        或：拆分大任务为更小单元

7天完成率 > 95%       → 弹出鼓励："进度超预期！"
                        建议：可以适当增加难度
                        或：提前进入下一阶段

连续3天未打卡          → 推送关怀消息（非催促）：
                        "休息也是成长的一部分，
                         随时回来，星空还在等你"

阶段截止日临近         → 进度诊断：
                        "距离基础阶段结束还有14天，
                         当前完成率62%，
                         建议每天多投入30分钟"

目标即将到期           → 预警：
                        "距离四级考试还有30天，
                         词汇掌握率78%，
                         建议增加每日单词量到80个"
```

### 三个预设模板

**模板一：考研倒计时规划**
- 适用：准备考研的大三/大四学生
- 内容：自动按距考试天数分配三阶段，科目+时间分配+每日任务模板
- 亮点：每周自动生成"距考研还有X天"倒计时提醒

**模板二：英语四六级冲刺**
- 适用：备考四六级的大一/大二学生
- 内容：30天/60天/90天三种冲刺方案，覆盖词汇+听力+阅读+写作+翻译
- 亮点：每日词汇量自动递增（第1周30个→第4周60个），模拟真题计时

**模板三：学期GPA提升计划**
- 适用：想提升绩点的任何年级学生
- 内容：按已选课程自动分配复习权重（低分课程权重更高），期中/期末前自动加密复习
- 亮点：与日程模块的课表数据联动，自动识别考试周并生成冲刺计划

## 社交激励方案

### 设计原则

**隐私优先、正向为主、压力可控。** 所有社交功能默认关闭，由用户主动开启。绝不公开展示失败或落后信息。

### 功能分层

#### 层级一：轻量互动（MVP后期或V2初期）

**好友互相查看**
- 可见信息：今日完成率（百分比）、连续打卡天数、当前等级
- 不可见信息：具体目标内容、任务详情（除非用户主动公开）
- 交互：点"加油"按钮 → 对方收到鼓励通知 + 双方各获5XP

```
┌─────────────────────────────────────┐
│ 好友动态                             │
│                                     │
│ 🔥 小明  今日 85%  连续32天          │
│    [加油 👍]                         │
│                                     │
│ 💪 小红  今日 100%  连续15天         │
│    [加油 👍]  "今日全勤！"            │
│                                     │
│ 😴 小华  今日 30%   连续2天          │
│    [加油 👍]                         │
│                                     │
│ ⏸️ 小李  冰冻中（考试周）            │
│    [加油 👍]                         │
└─────────────────────────────────────┘
```

**规则约束**：
- 每人每天最多给同一好友点赞1次（防刷）
- 好友列表复用社交伴侣模块的好友关系，不单独维护

#### 层级二：公开打卡（V2）

**打卡广播（可选）**
- 完成今日全部任务后，弹出选项："分享到校园墙？"
- 分享内容为模板化卡片，不泄露具体任务内容
- 示例："我今天完成了全部计划，连续坚持第32天！"
- 校园墙用户可点赞/评论，互动数据回流到规划模块

#### 层级三：集体目标（V3）

**使用场景**：
- 全班目标："软工2班全员通过四级" — 每个人的四级目标进度汇总为班级总进度
- 社团目标："跑团本月累计跑500公里" — 每个成员的跑步打卡累加
- 考研互助组："考研小分队冲冲冲" — 小组成员互相可见学习进度

**创建规则**：
- 任何人可创建集体目标，生成邀请链接/二维码
- 加入需确认，不能被强制拉入
- 个人详细数据仍然私密，集体目标只看汇总数字
- 集体目标完成后，所有成员获得额外成就："团队之星"

### 社交压力的平衡策略

| 风险 | 对策 |
|------|------|
| 看到别人进度好，自己焦虑 | 默认不显示排名，只展示"XX位好友今天完成了打卡" |
| 被强制拉入打卡群，压力大 | 所有社交功能opt-in，集体目标需手动确认加入 |
| 断签后不好意思面对好友 | 断签状态不通知好友，好友只看到"今日暂未打卡" |
| 过度比较导致目标扭曲 | "加油"是唯一社交操作，没有"催促"或"鄙视"按钮 |
| 社恐用户完全不想社交 | "隐身模式"：关闭所有社交可见性，纯单机使用 |

## MVP建议（功能清单 + 理由）

### MVP 核心三功能

| 序号 | 功能 | 理由 |
|------|------|------|
| **1** | **目标创建 + 两层任务拆解 + 进度追踪** | 这是规划模块的根基。没有目标体系，其他一切（游戏化、社交、AI）都无处附着。包含：创建目标、添加每日/每周任务、勾选完成、进度百分比显示。 |
| **2** | **每日打卡 + 连续/累计天数 + 补签机制** | 打卡是留存的命脉。没有每日打卡，用户没有理由每天回来。包含：今日任务清单、打卡完成动效、连续天数显示、每月2张补签卡、宽限期。 |
| **3** | **基础星空视觉 + 点亮动画** | 这是与所有竞品（滴答清单、Forest、番茄ToDo）的核心差异化。不需要完整星座系统，MVP只做：每个目标一颗星、完成时点亮动画、个人星空概览页。视觉冲击力是第一印象。 |

### MVP 功能细化清单

```
MVP 功能完整列表：

目标管理
  ├── 创建目标（标题、类型、截止日期、量化指标）
  ├── 4种目标类型：学业/技能/习惯/自定义
  ├── 目标状态：进行中/已完成/已暂停
  ├── 目标进度条（任务完成率自动计算）
  └── 目标列表页（按类型/状态筛选）

任务管理
  ├── 为目标添加任务（标题、日期、重复规则）
  ├── 任务类型：一次性/每日/每周
  ├── 今日任务清单（主页核心视图）
  ├── 勾选完成 + 微动效
  ├── 量化任务支持（50个单词、5公里等）
  └── 任务与日程的单向关联（可选"安排到日程"）

打卡系统
  ├── 连续天数 + 累计天数 双计数器
  ├── 每月2张补签卡
  ├── 宽限期（次日12:00前可补打前一天）
  ├── 今日全勤额外奖励
  └── 打卡日历（简单热力图，已打卡日高亮）

星空视觉
  ├── 个人星空概览页
  ├── 每个目标 = 一颗星
  ├── 4种星星状态（暗淡/闪烁/明亮/璀璨）
  ├── 完成目标时的点亮动画
  └── 总光芒值显示

基础数据
  ├── 今日/本周完成率显示
  └── 目标进度一览
```

### 明确不做的功能（后期迭代）

| 功能 | 推迟到 | 理由 |
|------|--------|------|
| AI 规划助手 | V2 | 需要 Prompt 工程调优 + Edge Function，MVP先验证核心循环 |
| 经验值/等级系统 | V2 | 打卡本身已提供反馈，XP系统是锦上添花 |
| 成就/徽章系统 | V2 | 先有足够用户数据再设计有意义的成就阈值 |
| 关键结果层(KR) | V2 | MVP 两层已够用，三层增加交互复杂度 |
| 好友互督/社交 | V2 | 社交依赖用户基数，MVP阶段用户太少没有网络效应 |
| 星座系统 | V3 | 需要用户累积多个目标后才有意义 |
| 集体目标 | V3 | 依赖社交功能和用户基数 |
| 成就卡片分享 | V3 | 依赖成就系统和校园墙联动 |
| 周报/月报/年报 | V2/V3 | 需要足够数据积累，至少使用2周后周报才有意义 |
| AI 动态调节 | V2 | 需要采集用户行为数据后才能做有意义的调节 |

### 上线时机建议

**推荐：在日程模块 MVP 稳定运行后推出，但设计和数据模型可提前并行准备。**

理由：

1. **技术依赖**：规划任务需要"安排到日程"的联动能力，如果日程 CRUD 不稳定，联动体验会很差。日程表的 `related_task_id` 字段（MODULE-SCHEDULE.md 已设计）是关键连接点。

2. **用户认知**：用户先理解"日程管时间"，再理解"规划管目标"，认知过渡更自然。同时上线两个相似模块容易让用户困惑"该用哪个"。

3. **并行准备**：日程开发期间，可以同步完成规划模块的数据库设计、UI 稿、核心算法。日程上线后立即进入规划的前端开发。

```
推荐时间线：

日程 MVP 开发中  →  规划数据库设计 + UI稿 + 算法设计（并行）
日程 MVP 上线    →  规划前端开发启动
日程 V2 开发中   →  规划 MVP 上线
                    规划 V2 开发（含AI助手 + 社交）
```

### 风险预警

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| 与日程功能边界模糊导致用户困惑 | 中 | 高 | MVP 阶段在 UI 上明确引导：日程="时间表"，规划="目标板"。导航入口、视觉风格做明显区分 |
| 游戏化机制设计过度导致开发周期膨胀 | 高 | 中 | 严格按优先级推进，MVP 只做星星+打卡，抵住加功能的冲动 |
| 星空视觉实现复杂度超预期 | 中 | 中 | MVP 用 CSS 动画 + SVG，不引入 WebGL/Canvas。视觉"够好"即可，不追求 3D |
| AI 规划质量不稳定 | 中 | 中 | V2 阶段先用模板化规划（非纯 AI 生成），AI 只做辅助调整，不做核心依赖 |
| 用户新鲜感过后打卡疲劳 | 高 | 高 | 补签机制 + 冰冻模式 + 回归奖励三重保障。关注"30天留存率"指标 |
