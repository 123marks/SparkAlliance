# OpenCode CLI — 星火规划模块头脑风暴指令

## 任务背景

星火校园（Spark Alliance）技术栈为 Vue 3 + TypeScript + Vite + Supabase。
已有模块：校园墙、AI助手、智能日程（含AI识别导入V2.5）、星火卡罗牌。

**新模块：星火规划（目标与任务）**
游戏化成长模块，核心链路：目标 → 任务拆解 → 打卡 → 积分/成就 → 社交分享。
重点关注：与日程模块的数据协同、游戏化数据的安全性、长期运营的可扩展性。

参考：
- `docs/project/MODULE-SCHEDULE.md`（了解日程模块设计，避免重复造轮子）
- 已有 `api_usage_logs` 表管理AI调用配额

## 你的任务

从**系统设计完整性、安全性与可扩展性**角度进行全面评审：

### 1. 与智能日程模块的协同边界

明确两个模块的数据归属，避免混乱：
- `schedule_events`（日程事件）与 `tasks`（规划任务）是什么关系？
  - 方案A：完全独立，互相引用（task_id 外键关联）
  - 方案B：tasks 是 schedule_events 的子集（一种类型）
  - 方案C：通过 Edge Function 同步，底层独立
- 用户删除日程中的事件，对应规划任务是否也应该删除？
- 跨模块查询（"我今天有什么规划任务要做"）如何优化性能？

### 2. 游戏化数据安全

游戏化系统的最大风险是数据作弊，评审以下问题：
- 打卡操作（habit_logs插入）是否需要服务端验证，防止时间戳伪造？
- 积分计算应该在前端还是服务端？（前端作弊风险 vs 性能开销）
- 成就解锁是否需要防重复触发？（触发器 + UNIQUE约束设计）
- 如果使用排行榜，如何防止刷分/排行污染？

### 3. 数据量与性能评估

规划模块是高频读写场景，评估以下风险：
- `habit_logs` 表：每个用户每天可能写入多条，一年后数据量？需要分区或归档吗？
- `tasks` 表：一个目标可能有100+子任务，列表查询如何优化？（游标分页？）
- 打卡热力图查询（SELECT过去365天记录）：索引设计？
- 社交场景：查询"我关注的好友今日打卡情况"如何设计索引？

### 4. AI规划配额整合

项目已有多个AI功能：
```
feature='ai_schedule_import'  → 每日5次
feature='tarot_reading'       → 每日3次
feature='goal_planning'       → 每日3次（建议）
```
- 三个AI功能共用 `api_usage_logs` 表，总AI调用成本评估？
- 是否需要设置跨功能每日总上限（如用户当日AI总调用不超过10次）？
- AI规划结果（任务拆解JSON）是否缓存？（同类型目标可复用模板）
- 目标规划AI调用时机：创建时一次性调用，还是支持重新生成？

### 5. 可扩展性设计

评估以下扩展场景的可行性：
- **会员功能**：付费用户可创建更多目标（免费限制3个？），如何在数据库层面控制？
- **模板市场**：用户可以发布自己的目标模板供他人使用，需要哪些额外表？
- **班级/社团集体目标**：多人共享一个目标，数据结构如何变化？
- **外部数据接入**：从微信运动/Keep导入健身数据自动打卡，预留什么接口？

### 6. 路由与模块接入

- 规划模块路由：`/app/planner` 还是 `/app/goals`？
- 子路由设计建议（首页/目标详情/习惯/成就/星空）
- 当前 `router/index.ts` 是否已有预留位置？（请检查现有路由）
- 底部导航栏是否需要为规划模块新增入口？（当前有几个Tab？）

### 7. 通知与提醒系统

- 每日打卡提醒通过什么实现？（Supabase Cron Job？前端本地通知？）
- 好友打卡后的"@你"通知，走哪个通道？
- 目标即将截止（3天内）的预警通知设计
- 通知系统是否需要独立的 `notifications` 表？还是复用现有机制？
