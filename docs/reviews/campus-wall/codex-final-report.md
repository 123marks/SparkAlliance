# Codex 最终综合评审报告  Spark Alliance

## 元评估：各工具评分校准

| 工具 | 原始评分 | Codex校准评分 | 校准原因 |
|------|---------|-------------|---------|
| Qoder | 62 | 58 | 对产品方向、视觉完成度和数据壁垒判断有价值，但对工程债、假数据占比、文案透支的惩罚略轻，整体稍偏宽松。 |
| Copilot | 71 | 60 | 代码问题抓得准，尤其是 XSS、状态语义和 token 混乱；但 71 分高估了当前“可上线程度”，因为核心功能仍以 mock 为主。 |
| iflow | 58 | 57 | 最接近当前真实状态。它准确抓住了首用引导缺失、信息过载、Soon 标签灾难和死链问题，只是对底层工程风险着墨稍少。 |
| OpenCode | 38 | 49 | 架构警报很有价值，尤其是对“本地授权方案”和后续微服务复杂度的判断；但把长期架构代价过多折算进当前得分，偏严苛。 |

## 综合评分：57/100（Codex最终裁定）

这个项目已经不是“空白概念”，而是一个**视觉完成度较高、认证流程初步可用、但核心价值尚未真正交付的高保真前端原型**。  
低于 50 分不客观，因为它确实有品牌调性、可运行路由、Supabase Auth、可构建前端壳；高于 60 分也不客观，因为 AI / 校园墙 / 个人中心都还没有形成可信的真实闭环，且规划与实现存在明显断层。

## 高频共识问题（多工具共同指出）

1. **核心功能大多停留在 UI Shell + Mock 数据阶段。**  
Qoder、Copilot、iflow、OpenCode 都明确指出：当前“已上线”更多是展示层，不是业务闭环。

2. **项目范围过大，MVP 失焦。**  
四份报告虽然措辞不同，但都在反对“校园超级 App”式大而全推进，认为当前阶段应收敛到少数高价值场景。

3. **信息架构和导航策略在透支用户信任。**  
App 首页的功能堆叠、侧栏大量 `Soon`、多个未交付入口、死链按钮，这一点在 Qoder、Copilot、iflow 中高度一致；我审代码后确认属实。

4. **可访问性与交互基础存在明显硬伤。**  
`frontend/src/style.css` 的全局 `cursor: none`，以及移动端侧栏状态语义反转，至少被 Copilot、Qoder、OpenCode 明确指出；源码验证无误。

5. **当前规划中的 AI 接入方案不可信或不可落地。**  
Qoder 与 OpenCode直接指出“本地授权”方案可行性存疑；我结合文档与代码后同意这一判断，而且官网与 Docs 还在持续放大这一承诺。

## 有价值的独特洞察

1. **iflow：首用路径不是“弱”，而是“零引导即丢弃”。**  
这一点非常关键。问题不只是功能未完成，而是新用户注册后立刻面对假数据、Soon 模块和 API Key 门槛，价值没有被建立。

2. **Qoder：真正可积累的数据壁垒是“课程评价/选课决策数据”，不是功能数量。**  
这比泛泛而谈“做大社区”更重要，因为它指向了可复利的数据资产。

3. **OpenCode：星火资讯 / 星火人才不是“再多写几个页面”，而是独立服务级复杂度。**  
这提醒团队必须把“产品想做”与“当前架构能承载”分开判断。

## 矛盾观点仲裁

### 1. 先做 AI Chat，还是先做校园墙闭环？

- Qoder 倾向先把 AI Chat 做真。
- Copilot 更倾向先交付一个真实闭环，建议优先校园墙。
- OpenCode认为校园墙是当前最容易落地的真实模块。

**Codex裁定：**
先做**“AI 接入策略纠偏”**，再优先落地**校园墙真实闭环**。  
原因是当前 Chat 页依赖的“本地授权”叙事本身就站不住，直接在错误前提上加功能只会继续返工；而校园墙更容易在 1 周内做出真实价值闭环，能最快把产品从“演示”推进到“可用”。

### 2. 要不要继续按“校园超级 App”推进？

**Codex裁定：不要。**  
从文档到官网再到应用实现，三者已经出现严重错位。当前阶段最应该做的是把项目重新定义成**AI 驱动的校园学习与选课决策助手**，社区只保留为轻量增强层，而不是主线。

### 3. 星火人才与星火资讯要不要押注？

**Codex裁定：**
- **Spark Talent 保留，但降到 P3 战略项。**
- **星火资讯从主路线拿掉，至少 6 个月内不要押注。**

原因：Talent 虽重，但与学生群体的职业跃迁高度相关，长期有商业化潜力；资讯则是最重、最散、最容易陷入无差异竞争的模块。

## Codex独立判断

### 根因分析（3个核心问题）

#### 1. 项目定位失真：规划、官网、应用三套叙事没有对齐

- 文档仍在定义“校园超级 App”，并规划校园购物、社交伴侣、资讯爬虫、人才匹配等大模块；
- 官网在宣传“公测”“在校用户”“满意率”“会员计划”“无限次答疑”；
- 应用内真实可用的只有登录壳和大量 mock UI。

这不是“还没做完”的正常差距，而是**定位与交付严重脱节**。它会先伤害信任，再伤害决策效率。

#### 2. 没有可信的核心闭环，导致产品价值无法被感知

我审查代码后确认：

- `frontend/src/pages/app/Chat.vue` 使用 `v-html` 渲染 mock 内容，保存配置靠 `localStorage`，回复靠 `setInterval` 模拟打字；
- `frontend/src/pages/app/CampusWall.vue` 的发帖和点赞只改前端内存状态；
- `frontend/src/pages/app/Profile.vue` 的热力图、成长进度、统计数据都是伪数据或随机数据；
- `frontend/src/pages/app/AppHome.vue` 首页大面积展示的是未兑现模块和假数据。

这意味着用户不会形成“解决问题 -> 得到结果 -> 愿意回来”的核心体验。

#### 3. 前端架构还停留在原型阶段，无法承接真业务

- 页面内同时混合 UI、状态、mock 数据和业务逻辑；
- 没有 store / service / types 的分层骨架；
- 没有 lint、test、CI；
- 路由守卫每次跳转都同步拉 session；
- 旧页面与新页面并存，增加认知和维护噪音。

这类架构不是“以后再整理”，而是会直接抬高每个真实功能落地的成本。

### 最大潜力点

**最大潜力不是“校园社交”，而是“AI + 选课/学习决策 + 校园场景数据”的组合。**

原因很直接：

- 大学生对高频刚需最敏感的是学习效率、选课避坑、实习过渡；
- 通用 AI 很强，但不懂具体学校、课程、老师、校内情境；
- 一旦积累课程评价、学长经验、学习行为和问答上下文，项目就开始拥有自己的数据护城河。

### 最值得押注的方向

**押注方向：做“校园学习决策 Copilot”，不是做“校园版微信”。**

建议的主线产品应是：

1. **AI 学习助手**  
解决题目、知识点、学习计划、论文润色等高频需求。

2. **AI 选课助手**  
以课程评价、教师风格、成绩分布、学长经验为核心数据资产。

3. **校园墙作为轻社区层**  
承接问答、吐槽、经验交换和轻量交易信息，不再承担“超级社交平台”的野心。

4. **Spark Talent 作为远期延伸**  
当学习和课程数据足够丰富后，再延伸到实习求职与能力名片，逻辑才顺。

## 优先级行动矩阵

### 🔴 P0 立即修复（影响核心体验，本周内）

1. **停止过度承诺。**  
下线或改写官网与 Docs 中所有超前承诺：`本地授权`、`在校用户 1200+`、`98% 满意率`、`会员计划`、`无限次答疑`、未交付付费能力。

2. **隐藏未交付入口与死链。**  
移除或折叠侧栏 `Soon` 模块、首页功能矩阵中的未上线能力、Profile 无响应设置项、首页“添加”死按钮。

3. **把 AI 接入策略从错误前提上纠偏。**  
废弃“检测用户本地登录 DeepSeek/豆包并授权”的 Web 方案，改成：
   - 服务端代理 / Edge Function
   - 平台统一额度
   - 或用户自填 API Key + 试用模式

4. **交付一个真实闭环，优先校园墙。**  
最小闭环必须包含：发帖、持久化、点赞、基础审核/举报。

5. **修掉明显基础风险。**  
去掉全局 `cursor: none`、修复 `v-html` 安全边界、拆分移动端侧栏状态语义。

### 🟠 P1 近期完成（1个月内，显著提升质量）

1. **建立前端工程骨架。**  
引入 Pinia、services、types、统一 toast/loading/error、ESLint、Vitest、CI。

2. **重构首页与首用路径。**  
把首页压缩成“欢迎 + 最近任务/最近对话 + 一个核心入口”，新增 2-3 步 onboarding。

3. **让 Profile 只展示真实数据。**  
热力图、成长值、学习目标、统计卡片要么接真数据，要么先隐藏。

4. **统一设计系统。**  
清理旧页面、统一 token 命名、用 SVG 图标替代 Emoji、梳理可复用基础组件。

### 🟡 P2 中期规划（3个月内，功能扩展）

1. **做 AI 选课助手 MVP。**  
优先建设课程评价、教师标签、学长经验、简单推荐逻辑。

2. **打通 AI 学习助手的真实能力。**  
支持真实对话历史、流式响应、图片题目输入、知识点总结。

3. **建立数据与增长看板。**  
追踪注册、首聊、首帖、7日留存、课程评价贡献数。

### 🟢 P3 长期布局（6个月+，战略方向）

1. **把 Spark Talent 作为独立战略线推进。**  
前提是先沉淀学生侧能力、项目与行为数据。

2. **若未来重启资讯，必须独立服务化。**  
不要再把星火资讯当成普通前端页面扩展；它是单独的内容基础设施项目。

3. **从“功能平台”升级为“数据平台”。**  
长期真正值钱的是课程、学习、能力、就业四类结构化校园数据。

## 给开发者的最终建议

现在最重要的不是继续加新模块，而是**立刻停止“超级 App”式扩张，收回承诺，先把 AI 接入策略改对，再把校园墙或 AI 学习助手做成一个真实、可信、能留住人的闭环**。只要核心闭环还是假的，越往后开发，返工和信任成本都会越高。
