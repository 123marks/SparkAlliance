# OpenCode CLI — 星火卡罗牌功能头脑风暴指令

## 任务背景

星火校园（Spark Alliance）是面向大学生的综合平台，技术栈为 Vue 3 + TypeScript + Vite + Supabase。
已有模块：校园墙（评论/举报/媒体上传）、AI助手（Chat）、智能日程（含AI识别导入）。

**新功能：星火卡罗牌**
融合塔罗文化与AI解读的趣味占卜模块，核心链路：抽牌 → AI解读 → 分享/记录。

参考已有 AI 配额管理方案：`api_usage_logs` 表（feature字段区分不同功能）

---

## 你的任务

从**系统设计完整性、安全性与可扩展性**角度进行全面评审：

### 1. 系统设计完整性

- [ ] 卡牌数据（`tarot_cards` 表）初始化方案：谁来写入78张牌的数据？管理后台？SQL seed脚本？
- [ ] 每日限额机制：复用 `api_usage_logs` 表是否合理？会有并发竞争条件（Race Condition）吗？
- [ ] 抽牌的"随机性"是否需要服务端保证？（防止前端伪造抽牌结果？）
- [ ] 三牌阵（过去/现在/未来）和单牌抽取，数据存储结构有何不同？

### 2. 安全与内容审核

- AI解读内容如何防范输出不当内容？（用户可能输入敏感问题）
  建议：Prompt 中明确添加内容边界声明，拒绝回答涉及伤害/违法内容
- 用户提问内容是否需要入库前过滤？（防止 SQL 注入 / XSS）
- 占卜内容是否有合规风险？（迷信相关法规，尤其面向未成年人的提醒）

### 3. AI 配额与成本控制

当前项目已有 `api_usage_logs` 表管理 AI 调用次数：
```
feature='ai_schedule_import' → 每日5次
feature='tarot_reading' → 每日3次（建议）
```

- 卡罗牌AI解读与日程AI识别共用一套限额逻辑，如何保证两者互不干扰？
- 如果用户当日日程识别已用3次、卡罗牌也用3次，总AI调用成本是否可控？
- 是否需要全局每日AI调用总上限（跨功能合计）？

### 4. 卡牌数据管理（后台预留）

- `tarot_cards` 表后续如何支持：
  - 管理员上传/更新卡牌图片？
  - 增加校园专属主题卡组（如"期末运势"限定卡组）？
  - 节日限定卡（春节/元宵/五四青年节特供）？
- 建议在 `tarot_cards` 表增加哪些字段来支持以上扩展？

### 5. 与现有模块整合评估

- **校园墙（CampusWall）**：分享抽牌结果到校园墙时，需要在 `posts` 表增加什么字段或关联？
- **AI助手（Chat）**：用户在聊天中说"帮我抽一张塔罗牌"，Chat模块如何调用卡罗牌逻辑？
- **智能日程（Schedule）**：抽到"学业运势"类卡牌时，如何一键关联到日程？需要什么跳转参数？
- **个人中心（Profile）**：展示"最近7天运势"摘要，数据如何聚合查询（性能）？

### 6. 路由与模块接入

- 卡罗牌的路由应该是 `/app/tarot` 还是 `/app/fortune`？
- 当前 `router/index.ts` 是否已有预留位置？（检查现有路由列表）
- 卡罗牌是否需要独立页面，还是可以作为一个 Modal/抽屉嵌入首页？

### 7. 可扩展性（未来版本规划）

卡罗牌功能未来可扩展方向，请评估可行性：
- 好友互占（A为B抽牌，AI解读B的问题）
- 群体占卜（社团活动前全员抽牌，AI给出集体运势）
- 占卜历史与AI趋势分析（"你最近一个月的整体运势趋势"）
- 付费卡组（高级主题牌面，会员解锁）
